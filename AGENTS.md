# Agent Instructions

This repository packages `@zktx.io/walrus-sites-preview`, a small CLI for
previewing already deployed Walrus Sites locally.

## Product Positioning

- Treat this project as a lightweight local review tool for deployed Walrus
  Sites.
- Do not describe it as a replacement for Walrus portals or existing deployment
  pipelines.
- When editing user-facing copy, preserve the tone that this tool sits alongside
  the established Walrus workflow.

## Source Layout

- `preview.mjs` is the published CLI entrypoint.
- `portal-worker/` and `portal-common/` are vendored source directories adapted
  from the upstream Walrus Sites portal.
- `dist/` is generated by `npm run build` and is included in the npm package.
- `scripts/sync-walrus-upstream.mjs` downloads a GitHub archive and syncs
  upstream source.
- `scripts/verify-maintenance.mjs` runs the maintenance verification suite.

## Upstream Maintenance Workflow

Use this flow when refreshing vendored Walrus Sites portal code:

1. Download upstream source into the ignored cache directory:

    ```sh
    npm run upstream:download
    ```

    This downloads a fresh GitHub archive and extracts it into:

    ```text
    .upstream/walrus-sites
    ```

    The directory is emptied before each download.

2. Replace vendored source from the cached upstream copy:

    ```sh
    npm run upstream:sync
    ```

    The sync maps:

    ```text
    portal/worker  ->  portal-worker
    portal/common  ->  portal-common
    ```

3. Run verification:

    ```sh
    npm run verify
    ```

For a one-command refresh and verification:

```sh
npm run upstream:update
```

To pin a branch, tag, or commit:

```sh
npm run upstream:update -- --ref <branch-tag-or-commit>
```

## Safety Rules

- The upstream cache must stay under `.upstream/`, which is ignored by git.
- The upstream cache is disposable. Do not treat it as a persistent git clone.
- Do not overwrite local changes in `portal-worker/` or `portal-common/` unless
  the user explicitly approves `--force`.
- After syncing upstream code, inspect the git diff before making follow-up
  changes.
- If upstream changes break the build, adapt the local preview CLI or webpack
  configuration deliberately. Do not blindly rewrite `preview.mjs`.
- Do not edit generated `dist/` files manually. Run `npm run build`.

## Verification

Run this before finishing maintenance changes:

```sh
npm run verify
```

It performs:

- `npm run format:check`
- `npm run lint`
- `npm run build`
- `npm pack --dry-run --ignore-scripts`

If a narrower change does not touch build or vendored code, at minimum run:

```sh
npm run format:check
npm run lint
```
